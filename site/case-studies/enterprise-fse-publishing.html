Enterprise FSE Publishing Platform

Governed content models and capabilities in a plugin; a swappable Full Site Editing block theme for presentation — so editorial structure survives a redesign.

  • WordPress · Full Site Editing
  • Domain policy with tests
  • Status: active development

Problem

WordPress projects routinely fuse content structure with the theme that renders it. Custom fields, post relationships, and editorial rules end up defined inside presentation code, so a redesign or a theme swap risks the data model — and the governance around who may publish what quietly erodes.

Approach

The platform draws a hard line between domain and presentation. A plugin owns the governed content models, editorial capabilities, and validation — the parts that must stay stable regardless of how the site looks. A Full Site Editing block theme owns presentation only, and is designed to be replaced without touching the domain layer.

What is built

  • Content models expressed as domain policy — registration, fields, and relationships — rather than incidental theme code.
  • Editorial capabilities mapped explicitly, so authoring and publishing rights are enforced at the domain boundary.
  • A swappable FSE block theme that consumes the models for presentation and can be exchanged without data migration.
  • Automated tests covering the domain policy, so the rules that protect editorial structure are verified rather than assumed.

Boundaries

  • This is a reference implementation in active development, not a turnkey product or a finished client site.
  • Content models and capability maps are illustrative of the pattern; a production deployment would tailor them to a real editorial team.
  • No production data, traffic figures, or client outcomes are claimed — the evidence is the runnable code and its tests.
