Resilient WooCommerce Commerce System

A replay-safe, idempotent webhook inbox and oversell-safe stock reservations — the workflows that keep revenue correct when networks and retries do their worst.

  • WooCommerce · WordPress
  • Integration-tested workflows
  • Status: active development

Problem

Payment gateways and fulfilment systems retry webhooks, connections drop mid-request, and shoppers race for the last unit in stock. Each of these turns into a revenue defect: an order processed twice, a refund applied again, or more items sold than exist. The failure is rarely in the happy path — it hides in the retries and the concurrency.

Approach

The system treats every inbound event as potentially duplicated and every stock change as contended. Webhooks flow through an inbox that records each event by its identifier and refuses to act on the same one twice. Stock is protected by reservations that decrement atomically, so two simultaneous checkouts cannot both claim the final unit.

What is built

  • An idempotent webhook inbox that deduplicates by event identifier, making replays and retries safe by construction.
  • Oversell-safe stock reservations with atomic decrements, so concurrent checkouts settle to a correct total.
  • Integration tests that reproduce duplicate delivery and concurrent contention, then assert the invariants hold.

Boundaries

  • This is a reference implementation in active development, not a drop-in plugin for an existing storefront.
  • Gateway and fulfilment integrations are illustrative of the pattern rather than certified against a specific provider.
  • No live sales, conversion, or revenue figures are claimed — the evidence is the tested workflow behaviour under failure.
